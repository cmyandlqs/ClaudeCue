# 回焦问题分析与修复方案

## 1. 问题背景

ccCue 的核心交互之一是：

1. Claude 触发事件 -> 弹出通知。
2. 用户点击通知（或使用快捷键）-> 快速回到对应终端窗口。

当前现象是“通知能弹出，但回焦经常失败/无效”。

另外出现两个相关问题：

1. 缺少可用日志，难以定位失败原因。
2. `python -m notifier.main` 在 PowerShell 中 `Ctrl+C` 退出不稳定。

## 2. 已观察到的现象

1. 通知显示正常，点击后终端不一定到前台。
2. 多终端并行时更容易失败或命中错误窗口。
3. 控制台中能看到启动日志，但缺少回焦链路细节。
4. `QSystemTrayIcon::setVisible: No Icon set` 警告出现过。

## 3. 根因拆解

### 3.1 窗口识别与会话关联不够可靠

1. 仅靠标题关键字 + 局部类名容易漏识别。
2. notifier 侧“事后绑定当前前台窗口”的时机可能已经偏离真实 Claude 终端。
3. 多终端场景下，`session_id -> hwnd` 映射会抖动或被污染。

### 3.2 Windows 前台切换受系统策略限制

1. 后台进程直接 `SetForegroundWindow` 常被系统拦截。
2. 不同机器/权限/UAC 状态下表现不一致。
3. 需要组合调用和降级手段，而不是单 API。

### 3.3 缺少可观测性

1. 之前日志不足以回答“为什么这次没切过去”。
2. 无法区分“候选窗口没匹配到”与“API 被系统拒绝”。

## 4. 当前已落地的改进（代码已实现）

### 4.1 日志系统

1. 新增文件日志：`logs/notifier.log`（滚动日志，保留多份）。
2. 控制台日志保留 INFO；文件日志写入 DEBUG 细节。
3. 回焦链路已记录：
   - 是否命中 `terminal_hwnd_hint`
   - 是否命中 `session_id` 映射
   - 候选窗口评分（标题、类名、句柄）
   - 最终前台窗口句柄和成功标记

### 4.2 回焦策略增强

1. hook 侧新增 `terminal_hwnd_hint`（事件触发瞬间采样前台窗口句柄）。
2. notifier 侧优先使用 `terminal_hwnd_hint`，再回退到本地映射/候选评分。
3. 前台切换采用组合策略：
   - `ShowWindow`
   - `BringWindowToTop`
   - `AttachThreadInput`
   - ALT key hack
   - `SetForegroundWindow`
   - TopMost 开关兜底

### 4.3 快捷键交互

1. 新增全局热键：`Ctrl + Alt + Space`
2. 热键触发“按最近通知事件回焦终端”，不依赖鼠标点击弹窗。
3. 弹窗内新增快捷键提示文案。

### 4.4 退出行为

1. 增加 SIGINT/SIGBREAK 处理与 Qt 信号轮询，支持 `Ctrl+C` 优雅退出。
2. 退出时会停止 server/timer，并注销全局热键。

### 4.5 托盘稳定性

1. 托盘图标设置默认系统图标，减少 `No Icon set` 告警。

## 5. 如何排查“仍然无法回焦”

## 5.1 先确认版本生效

1. 完全退出旧 notifier 进程。
2. 重新启动：`python -m notifier.main`
3. 触发一条通知后点击，或按 `Ctrl+Alt+Space`。

## 5.2 查看日志

日志文件：`ccCue/logs/notifier.log`

重点关键词：

1. `Hotkey pressed, trying focus with latest event`
2. `Trying hinted hwnd=...`
3. `Trying mapped session hwnd=...`
4. `Candidate hwnd=... score=...`
5. `Focus result hwnd=... foreground=... ok=...`
6. `No terminal window could be focused ...`

基于日志可判断：

1. 是否拿到正确的 `hwnd hint`
2. 是否窗口已失效（句柄无效/不可见）
3. 是否系统拒绝前台切换

## 6. 下一步可选增强（若当前仍不稳定）

1. 引入进程级匹配：
   - 把 hook 触发进程 PID、父进程 PID 一并传给 notifier
   - 然后按 PID 反查并锁定目标窗口
2. 引入“窗口快照缓存 + 失效清理”：
   - 定时剔除失效 hwnd，避免旧映射干扰
3. 增加失败降级反馈：
   - 回焦失败时托盘提示“已定位终端，请手动点任务栏图标”
4. 增加诊断命令：
   - 一键导出最近 N 次回焦尝试与候选评分，便于快速复盘

## 7. 当前交互建议

在你现有使用方式上，推荐主交互为：

1. 通知弹出时优先按 `Ctrl+Alt+Space`（比鼠标点击更稳定）。
2. 若未成功，查看 `logs/notifier.log` 对应时间段日志并反馈。

这样可以在不改动主流程的前提下，快速迭代到稳定回焦。

## 8. 两个关键问题的结论

### 8.1 Windows 下是否可以定位所有窗口？窗口是否有唯一 ID？

结论：可以定位大部分顶层窗口；每个窗口有句柄 `HWND` 作为“当前生命周期内唯一标识”。

说明：

1. 可以通过 `EnumWindows` 枚举系统当前所有顶层窗口。
2. 每个窗口有：
   - `HWND`（窗口句柄）
   - 所属进程 PID / 线程 TID
   - 标题（WindowText）
   - 类名（ClassName）
3. `HWND` 在该窗口存活期间是唯一的，但不是永久 ID：
   - 窗口关闭再重建后，`HWND` 会变化。
   - 因此不能把 `HWND` 当长期稳定主键存档使用，只适合短时会话映射。

工程含义：

1. 可以做“事件发生后短时回焦”（几秒到几分钟）；
2. 若要更稳，需要联合 `PID + 类名 + 标题特征 + 时间窗口` 做二次校验。

### 8.2 能否用类似 Alt+Tab 的方式切到指定窗口并置顶？

结论：可以实现“等价目标”，但不是直接调用系统 Alt+Tab UI 本身。

可行方案：

1. 直接按窗口句柄切换（推荐）
   - `ShowWindow`
   - `BringWindowToTop`
   - `SetForegroundWindow`
   - 必要时 `AttachThreadInput` + TopMost toggle
2. 这是当前项目已经在做的主路径，理论上可达到“像 Alt+Tab 一样切过去”的效果。

边界条件：

1. Windows 有前台焦点保护策略，后台进程抢焦点可能被拒绝。
2. 部分场景会出现“API 返回成功但体感未变化”：
   - 目标窗口本来就在前台；
   - 命中了错误窗口（同类终端多开）；
   - 切到了目标窗口但被其他窗口瞬时抢回。

工程结论：

1. “按指定窗口切换并置顶”是能做到的；
2. 关键不在“能不能调用”，而在“目标窗口识别是否稳定 + 焦点保护策略的规避是否完整”。

## 9. 基于当前日志的额外判断

你提供的日志显示：

1. `Failed to register global hotkey Ctrl+Alt+Space`
   - 说明热键被系统或其他软件占用，当前热键功能未生效。
2. `Focus result ... ok=True`
   - 说明系统层面这次 `SetForegroundWindow` 结果判定为成功。
   - 但仍体感“没切过去”，更可能是“命中了不对的窗口”而不是“API 调用失败”。

因此下一步应优先做：

1. 热键冲突处理（支持可配置热键与冲突回退）；
2. 窗口目标识别升级（从仅 `HWND hint` 升级为 `HWND + PID + 标题特征` 复核）；
3. 在日志里增加“命中窗口详情快照”，避免只看到 `hwnd` 数字而无法人工判断是否命中正确窗口。

## 10. 本轮已实施更新（2026-04-10）

### 10.1 热键冲突回退

已将全局热键改为候选列表自动注册，按顺序尝试：

1. `Ctrl+Alt+Space`
2. `Ctrl+Alt+Q`
3. `Ctrl+Shift+Space`

成功后会在日志中打印“实际生效热键”，并同步展示在通知卡片底部提示中。

### 10.2 回焦目标升级为多维复核

hook 事件现在会额外上报：

1. `terminal_hwnd_hint`
2. `terminal_pid_hint`
3. `terminal_title_hint`
4. `terminal_class_hint`

notifier 侧使用策略：

1. 先尝试 `HWND`，但必须通过 `PID/类名/标题` 复核；
2. `HWND` 不可用时按 `PID` 枚举候选窗口并评分；
3. 再回退 session 映射与通用候选评分。

### 10.3 日志可读性增强

回焦日志不再只输出 `hwnd`，会输出窗口快照：

`hwnd + pid + class + title`

这可直接判断“到底命中了哪个窗口”，便于确认是否是“命中错误窗口”。

## 11. 最新现场结论（基于 2026-04-10 15:56~15:57 日志）

已确认不是“焦点 API 失效”，而是“目标窗口识别错误”：

1. 日志显示命中窗口为：
   - `class=Chrome_WidgetWin_1`
   - `title='智谱AI开放平台 - Google Chrome'`
2. 回焦链路返回 `ok=True`，说明系统确实切到了该窗口。
3. 因此体感“没有回到终端”是因为目标本身选错（选到了 Chrome）。

## 12. 针对性修复（已落地）

### 12.1 hook 侧 hint 过滤

hook 不再无条件上报前台窗口 hint；仅当前台窗口“看起来是终端”时才上报：

1. 类名命中：
   - `CASCADIA_HOSTING_WINDOW_CLASS`
   - `CascadiaHost`
   - `ConsoleWindowClass`
2. 或标题命中：
   - `Windows Terminal` / `PowerShell` / `Command Prompt` / `cmd` / `Claude`

效果：避免把 Chrome/IDE 等非终端窗口错误写入 hint。

### 12.2 notifier 侧 hint 二次校验

即使收到了 `terminal_hwnd_hint`，也先判断该窗口是否终端：

1. 非终端则直接跳过，不再盲目回焦。
2. 继续使用 PID/标题/类名复核与候选评分。

### 12.3 最近活跃终端缓存（fallback）

notifier 新增后台探测：

1. 每 300ms 采样一次当前活动终端。
2. 事件缺少有效 hint 时，回退到“最近活跃终端 hwnd”。

效果：即使你在事件触发瞬间切到浏览器，也可回到最近使用的终端。
