# ccCue 问题文档（现状与方案）

## 1. 文档目的

本文件用于沉淀当前仍需解决的问题，以及每个问题对应的解决方案与优先级，作为近期执行依据。

## 2. 当前问题总览

### P0：运行时一致性与安装成功率

问题：

1. 历史形态依赖用户本机 Python 与包环境，导致机器间成功率不一致。
2. 普通用户预期是“装完即用”，但环境差异会造成启动失败或功能缺失。

方案：

1. 已决策采用独立可执行文件路线（方案 B）。
2. 安装器不再依赖用户预装 Python/PySide6/pywin32。
3. hooks 调用链改为安装目录内可执行产物。
4. 安装后强制执行自检（hooks 写入、notifier 健康检查）。

验收：

1. 全新环境安装后可直接收到通知。
2. notifier 可自动拉起并通过健康检查。

### P1：发布流程与对外交付规范不足

问题：

1. 发布过程缺少统一检查清单，容易依赖人工经验。
2. 用户侧变更说明、已知问题、回滚说明不够标准化。

方案：

1. 建立最小发布清单：版本号、安装包、变更摘要、已知问题、回滚步骤。
2. 将构建、验收、发布动作标准化，确保每次发布路径一致。

验收：

1. 任一发布可按清单执行并可复盘。
2. 版本问题可快速定位到“构建/配置/运行时”层面。

### P1：用户文档不足，开发文档与用户文档混用

问题：

1. 历史文档偏设计与开发，不利于普通用户直接上手。
2. 用户遇到“通知不弹/回焦失败/权限请求无提示”时缺少按症状排查指引。

方案：

1. 保留统一需求文档与统一问题文档作为核心基线。
2. 后续补齐“项目改进文档”，承接具体改造计划与排期。
3. 文档统一 UTF-8 编码，消除历史乱码与重复描述。

验收：

1. 新用户无需阅读开发细节即可完成安装、验证与自助恢复。

### P1：安装路径与磁盘占用透明度不足（用户对 C 盘敏感）

问题：

1. 默认安装到 C 盘（`%LOCALAPPDATA%`）会引发部分用户对空间占用的担忧。
2. 若文档不明确“可改安装目录”和“预计占用大小”，用户会感知为强制占用 C 盘。

方案：

1. EXE 安装流程明确展示并支持修改安装目录（含 D 盘示例）。
2. CLI 文档默认给出带 `--target` 的示例（含 D 盘示例）。
3. 在需求文档/发布说明中固定披露：
   - 安装包大小
   - 安装后预计占用范围

验收：

1. 用户在安装前即可知道体积与默认路径。
2. 用户可无障碍安装到非 C 盘目录。

### P2：回焦稳定性在复杂窗口场景下仍有边界

问题：

1. 多终端、多应用切换时，可能出现目标识别偏差。
2. Windows 前台焦点策略会对后台抢焦点产生限制。

方案：

1. 继续使用多维识别（hwnd/pid/class/title）与候选评分。
2. 保留快捷键回焦与冲突回退机制。
3. 强化日志快照信息，便于判定“识别错误”还是“系统策略拦截”。

验收：

1. 回焦失败日志可定位根因，不出现“不可解释失败”。
2. 主流使用场景下回焦成功率稳定。

### P2：诊断信息对普通用户可读性不足

问题：

1. 虽有 `doctor/restore`，但用户难以快速判断下一步动作。

方案：

1. 诊断输出分级（PASS/WARN/FAIL）。
2. FAIL 必须给出可复制执行的修复命令。
3. 输出最小问题上报模板（版本、日志路径、诊断摘要）。

验收：

1. 用户可基于诊断输出独立完成大部分常见修复。

## 3. 已达成共识

1. 不恢复 wrapper 路线。
2. 运行时采用方案 B：独立可执行文件，不依赖用户 Python 环境。
3. 近期优先级顺序：P0 运行时一致性 > P1 发布与文档 > P2 诊断与体验增强。

## 4. 执行边界

1. 本文只描述问题与对应方案，不展开详细项目排期。
2. 详细实施计划放入《项目改进文档》，在需求文档与问题文档审核通过后编写。
