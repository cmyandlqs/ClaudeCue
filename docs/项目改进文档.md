# ccCue 项目改进文档

## 1. 文档目标

基于《需求文档》《问题文档》，给出可落地的改进计划与开发步骤，确保项目从“可用”进入“可稳定发布、可维护、可恢复”状态。

## 2. 范围与原则

1. 架构不变：坚持 `hooks -> bootstrap -> notifier`，不恢复 wrapper。
2. 运行时策略不变：采用方案 B，目标为“不依赖用户机器预装 Python/PySide6/pywin32”。
3. 交付目标：同时支持 EXE 安装与源码一键安装（CLI），且行为一致。
4. 质量门槛：每阶段都需通过 lint、死代码检查、自动化测试与安装回归。

1. 主分支最新提交已推送：`b221fa9`。
2. 文档已统一为三份：需求、问题、项目改进。
3. 静态检查现状：
   - `ruff` 通过
   - `vulture` 通过
4. 自动化测试现状：核心测试集 `38 passed`。

## 4. 里程碑规划

### M1（P0）运行时独立化与安装成功率

目标：完成独立可执行交付闭环，确保普通用户“装完即用”。

### M2（P1）发布流程产品化

目标：形成标准发布流水与可复盘交付清单。

### M3（P1）用户体验与文档闭环

目标：让用户不看开发文档也能安装、诊断、恢复。

### M4（P2）回焦与诊断增强

目标：提升复杂场景下回焦稳定性与问题定位效率。

## 5. 详细开发步骤

## 5.1 M1：运行时独立化与安装成功率（最高优先级）

### 步骤 1：定义打包边界与产物结构

1. 明确需要打包为可执行文件的模块：
   - `notifier` 主程序
   - `cli` 主程序
2. 明确 hooks 入口的调用方式：
   - hooks 只负责接收与转发，不依赖系统 Python 路径。
3. 输出产物目录规范：
   - `runtime/bin`（可执行文件）
   - `runtime/hooks`（hook 脚本或桥接入口）
   - `runtime/config`（配置模板）

交付物：`docs/` 内补充打包产物结构说明（可并入后续发布说明）。

### 步骤 2：实现可执行构建脚本

1. 在 `installer/` 增加统一构建入口（例如 `build_runtime.bat`）。
2. 固化构建参数（版本号、输出路径、是否清理旧产物）。
3. 产物命名包含版本，避免覆盖混淆。

交付物：本地一键构建命令 + 可执行产物。

### 步骤 3：安装器切换到“可执行产物模式”

1. `installer/ccCue.iss` 改为安装可执行产物，不再要求 Python 预检。
2. 安装完成后执行自检：
   - hooks 写入成功
   - notifier 健康端点可达
3. 保留安装目录可修改能力（默认 `%LOCALAPPDATA%\ccCue`，支持 D 盘）。

交付物：新版 EXE 安装包。

### 步骤 4：CLI 安装路径与 EXE 行为对齐

1. `python -m cli.main install --source . --target <dir>` 保持可用。
2. CLI 安装后的目录结构、hook 写入策略、回滚策略与 EXE 保持一致。
3. 增加安装后体积与路径提示（用于用户透明感知）。

交付物：CLI 安装回归通过记录。

### 步骤 5：M1 验收

1. 全新 Windows 环境安装成功（EXE 与 CLI 各一次）。
2. 首次 Claude 会话触发通知闭环成功。
3. 不依赖用户手动安装 Python 包。
4. 卸载后 Claude 配置可恢复。

## 5.2 M2：发布流程产品化

### 步骤 1：建立发布清单模板

1. 版本号规则（语义化版本）。
2. 必备产物：
   - EXE
   - 校验信息（哈希）
   - 变更说明
3. 已知问题与回滚说明模板。

### 步骤 2：发布前自动检查

1. 运行 `ruff`、`vulture`、`pytest`。
2. 运行安装自检脚本（静默安装 + `doctor`）。
3. 失败即阻断发布。

### 步骤 3：发布后验证

1. 干净环境安装验证。
2. 升级安装验证（旧版本升级到新版本）。
3. 卸载恢复验证。

## 5.3 M3：用户体验与文档闭环

### 步骤 1：补齐用户向说明

1. 5 分钟安装指南（EXE/CLI 双路径）。
2. D 盘安装示例与 `--target` 示例。
3. 安装包大小、安装后占用范围说明。

### 步骤 2：补齐常见问题（FAQ）

1. 不弹通知。
2. 回焦失败。
3. 权限请求未提醒。
4. 配置恢复与卸载失败。

### 步骤 3：补齐故障恢复手册

1. `doctor` 输出判读。
2. `restore` 与 `uninstall --purge` 使用时机。
3. 最小问题反馈模板。

## 5.4 M4：回焦与诊断增强

### 步骤 1：回焦识别稳定性增强

1. 加强多维候选评分（hwnd/pid/class/title/session）。
2. 对错误命中场景增加保护逻辑和日志标注。

### 步骤 2：诊断输出分级

1. `PASS/WARN/FAIL` 统一输出格式。
2. `FAIL` 必须附可复制修复命令。
3. 输出日志路径、版本、运行目录摘要。

### 步骤 3：问题采集闭环

1. 一键导出诊断摘要。
2. 发布模板化反馈字段，便于快速定位。

## 6. 任务拆分（按模块）

1. `installer/`：
   - 可执行构建脚本
   - Inno 安装逻辑更新
   - 安装后自检串联
2. `cli/`：
   - 安装日志与结果输出增强
   - 诊断分级输出
3. `hooks/`：
   - 可执行产物调用入口对接
   - 错误兜底与不阻塞保障
4. `notifier/`：
   - 回焦策略优化
   - 诊断日志可读性提升
5. `tests/`：
   - 新增独立运行时安装回归测试
   - 新增诊断输出分级测试
6. `docs/`：
   - 发布清单
   - 用户手册
   - 故障排查与恢复手册

## 7. 验收清单（统一 DoD）

1. 静态检查通过：`ruff`、`vulture` 无阻断问题。
2. 自动化测试通过：核心测试全绿。
3. EXE 安装可选目录（含 D 盘）可用。
4. CLI 安装可指定 `--target`，且与 EXE 行为一致。
5. 安装后自检通过（hooks + notifier health）。
6. 卸载后配置恢复通过。
7. 文档与实现一致，用户可按文档完成自助安装与恢复。

## 8. 风险与应对

1. 打包后体积上涨：
   - 应对：分离调试符号、压缩策略优化、披露体积变化。
2. 焦点策略系统差异导致回焦不稳定：
   - 应对：保留热键兜底、增强候选评分、补充日志证据。
3. 安装目录切换引入路径兼容问题：
   - 应对：统一路径解析函数，增加 C/D 盘安装回归。
4. 发布节奏与质量冲突：
   - 应对：发布前检查硬门槛，不满足不发布。
