# Claude Code 提醒助手第一阶段开发文档

## 1. 文档目标

本文档用于定义第一阶段的开发范围、系统结构、模块职责、接口约定、事件模型、配置设计与验证标准。

第一阶段只覆盖以下能力：

1. 启动并监听 Claude Code 运行过程
2. 识别关键提醒事件
3. 通过 Windows 顶置悬浮窗提醒用户
4. 在需要时将用户快速带回终端

本阶段明确不做：

1. 终端旁路历史导航窗
2. 对话节点结构化展示
3. 多终端复杂联动
4. 深度集成 Windows Terminal 内部 UI

## 2. 阶段目标

### 2.1 产品目标

当 Claude Code 在后台运行时，若出现以下情况，系统应主动提醒用户：

1. 任务已完成
2. Claude Code 需要用户继续输入
3. Claude Code 遇到权限批准或权限拦截
4. Claude Code 异常中断
5. Claude Code 长时间无输出，疑似等待或卡住

### 2.2 工程目标

第一阶段必须形成一个最小闭环：

1. 通过包装器启动 Claude Code
2. 监听输出并识别事件
3. 将事件发送给提醒程序
4. 弹出可用的顶置提醒窗
5. 用户点击后可以切回终端

## 3. 范围定义

### 3.1 In Scope

1. Claude Code 包装启动器
2. Claude Code 输出监听
3. 事件识别规则引擎
4. 本地进程间通信
5. Windows 桌面顶置提醒窗
6. 系统托盘常驻
7. 配置文件与基础日志

### 3.2 Out of Scope

1. 历史会话解析
2. 对话节点跳转
3. 云同步
4. 多设备通知
5. 复杂权限编排
6. 插件化规则市场

## 4. 技术选型

### 4.1 语言

使用 Python。

原因：

1. 适合快速做 PoC 与迭代
2. 适合处理进程、文本流、正则匹配
3. 便于构建桌面小工具

### 4.2 UI 框架

使用 `PySide6`。

原因：

1. 更适合长期维护的 Windows 桌面界面
2. 支持托盘、置顶、定时器、窗口控制
3. 未来扩展设置页、事件记录页更方便

### 4.3 进程通信

第一阶段建议使用本地 HTTP 接口。

原因：

1. 实现简单
2. 调试方便
3. 监听器与 UI 可独立启动和排查

备选：

1. 本地 socket
2. 命名管道

但第一阶段不优先。

### 4.4 配置格式

使用 `YAML` 或 `JSON`。

建议优先 `YAML`，因为可读性更高。

## 5. 系统总览

第一阶段系统由两个主进程组成：

1. `notifier-app`
   - 常驻桌面程序
   - 负责托盘、悬浮窗、提醒声音、终端聚焦
   - 提供本地事件接收接口

2. `cc-wrapper`
   - Claude Code 启动包装器
   - 负责启动 Claude Code、监听输出、识别事件
   - 向 `notifier-app` 发送标准化事件

## 6. 进程关系

### 6.1 运行关系

启动流程如下：

1. 用户先启动 `notifier-app`
2. `notifier-app` 在本地端口监听事件
3. 用户通过 `cc-wrapper` 启动 Claude Code
4. `cc-wrapper` 监听 Claude Code 输出
5. `cc-wrapper` 命中规则后向 `notifier-app` 发送事件
6. `notifier-app` 弹出提醒

### 6.2 异常关系

1. 若 `notifier-app` 未启动，`cc-wrapper` 应降级为仅本地打印警告和写日志
2. 若 `cc-wrapper` 异常退出，`notifier-app` 仍可保持运行
3. 若 Claude Code 正常退出但没有命中事件，也应记录退出状态

## 8. 核心模块设计

## 8.1 wrapper 模块

职责：

1. 启动 Claude Code
2. 获取标准输出与标准错误
3. 将输出流按行或按块送入规则引擎
4. 识别提醒事件
5. 向 `notifier-app` 发送标准事件

子模块建议：

### `launcher.py`

职责：

1. 组装 Claude Code 启动命令
2. 启动子进程
3. 保存 PID
4. 记录启动时间、工作目录、命令参数

### `stream_reader.py`

职责：

1. 分别读取 stdout / stderr
2. 处理缓冲与编码
3. 将内容转为统一的文本片段
4. 将片段交给检测器

### `event_detector.py`

职责：

1. 对输入文本做基础预处理
2. 维护当前会话状态
3. 调用规则引擎识别事件
4. 控制去重、节流和状态切换

### `rule_engine.py`

职责：

1. 基于关键词、正则和状态组合判断事件
2. 输出标准事件对象
3. 支持配置化规则

### `client.py`

职责：

1. 将事件发送给 `notifier-app`
2. 处理接口失败重试
3. 在失败时写本地日志

## 8.2 notifier 模块

职责：

1. 作为常驻程序运行
2. 接收 wrapper 发送的事件
3. 根据事件级别展示提醒
4. 管理托盘图标与基础设置
5. 支持点击提醒后聚焦终端

子模块建议：

### `server.py`

职责：

1. 提供本地 HTTP 服务
2. 接收 `POST /events`
3. 校验数据结构
4. 转交 UI 线程处理

### `overlay.py`

职责：

1. 渲染顶置悬浮窗
2. 根据事件级别切换样式
3. 控制自动消失或常驻
4. 处理点击行为

### `tray.py`

职责：

1. 展示系统托盘图标
2. 提供基本菜单
3. 支持退出、查看状态、打开日志

### `sound.py`

职责：

1. 根据提醒等级播放不同声音
2. 提供静音开关

### `window_focus.py`

职责：

1. 定位目标终端窗口
2. 尝试切换到对应窗口
3. 必要时回退为仅提示用户手动切换

## 8.3 common 模块

职责：

1. 统一配置读取
2. 统一日志格式
3. 统一事件模型
4. 统一常量和枚举

## 9. 事件模型设计

第一阶段所有模块之间只传“标准事件对象”，禁止直接传裸字符串。

### 9.1 事件类型

第一阶段定义以下事件类型：

1. `task_completed`
2. `needs_input`
3. `permission_blocked`
4. `error_detected`
5. `idle_timeout`
6. `process_started`
7. `process_exited`

说明：

1. `process_started` 与 `process_exited` 主要用于日志和后续扩展
2. 对用户实际提醒的重点是前五类

### 9.2 事件级别

定义以下严重级别：

1. `info`
2. `warning`
3. `critical`

建议映射：

1. `task_completed` -> `info`
2. `needs_input` -> `warning`
3. `permission_blocked` -> `critical`
4. `error_detected` -> `critical`
5. `idle_timeout` -> `warning`

### 9.3 事件对象结构

建议统一结构如下：

```json
{
  "event_id": "uuid",
  "event_type": "permission_blocked",
  "severity": "critical",
  "title": "Claude Code 等待权限批准",
  "message": "检测到权限批准或拦截，请返回终端处理。",
  "source": "cc-wrapper",
  "session_id": "20260410-090000-abc123",
  "process_id": 12345,
  "terminal_hint": {
    "window_title": "Claude Code",
    "cwd": "D:/project/demo"
  },
  "match": {
    "rule_id": "permission.keyword.01",
    "pattern": "approval",
    "sample_text": "..."
  },
  "timestamps": {
    "occurred_at": "2026-04-10T09:00:00+08:00",
    "sent_at": "2026-04-10T09:00:01+08:00"
  },
  "display": {
    "sticky": true,
    "play_sound": true,
    "timeout_ms": 0
  }
}
```

### 9.4 字段说明

1. `event_id`：唯一 ID，用于去重和追踪
2. `event_type`：标准事件类型
3. `severity`：严重级别
4. `title`：悬浮窗标题
5. `message`：悬浮窗内容
6. `source`：事件来源，第一阶段固定为 `cc-wrapper`
7. `session_id`：本次 Claude Code 会话 ID
8. `process_id`：Claude Code 进程 ID
9. `terminal_hint`：用于尝试聚焦窗口的辅助信息
10. `match`：命中规则信息，便于排错
11. `timestamps`：时间戳
12. `display`：建议展示方式

## 10. 状态模型设计

为了避免重复提醒，`event_detector` 需要维护最小状态机。

### 10.1 会话状态

建议定义以下内部状态：

1. `running`
2. `waiting_input`
3. `waiting_permission`
4. `idle_suspected`
5. `completed`
6. `exited`

### 10.2 状态切换原则

1. 启动后进入 `running`
2. 命中输入等待规则后进入 `waiting_input`
3. 命中权限规则后进入 `waiting_permission`
4. 超时未输出进入 `idle_suspected`
5. 命中完成规则后进入 `completed`
6. 进程退出后进入 `exited`

### 10.3 去重原则

同一状态下重复命中同类事件时，不应持续重复弹窗。

建议策略：

1. 相同 `event_type` 在 30 秒内只提醒一次
2. `critical` 事件可允许更短节流，但仍需防抖
3. 状态切换后允许再次提醒

## 11. 规则引擎设计

第一阶段不做复杂 NLP，只使用规则匹配。

### 11.1 规则来源

规则来源分两类：

1. 内置默认规则
2. 外部配置规则

建议先以内置为主，配置覆盖为辅。

### 11.2 规则结构

建议规则结构如下：

```yaml
- id: permission.keyword.01
  enabled: true
  event_type: permission_blocked
  severity: critical
  match_type: contains
  pattern: "approval"
  applies_to:
    - stdout
    - stderr

- id: task.regex.01
  enabled: true
  event_type: task_completed
  severity: info
  match_type: regex
  pattern: "(completed|finished|done)"
  applies_to:
    - stdout
```

### 11.3 支持的匹配方式

第一阶段建议支持：

1. `contains`
2. `regex`
3. `any_of`

不建议第一阶段支持：

1. 复杂条件组合 DSL
2. 跨多段输出的高级语义规则

### 11.4 第一阶段最低规则集

必须先覆盖以下事件：

1. 任务完成
2. 等待输入
3. 权限拦截或批准
4. 异常错误
5. 空闲超时

### 11.5 空闲超时规则

`idle_timeout` 不是文本匹配，而是时间规则：

1. Claude Code 进程仍存活
2. 最近 N 秒没有输出
3. 当前状态未完成、未退出

建议默认阈值：

1. 60 秒进入可疑空闲
2. 120 秒触发提醒

## 12. 本地接口设计

第一阶段建议由 `notifier-app` 提供本地 HTTP 服务。

## 12.1 服务地址

建议默认：

```text
http://127.0.0.1:45872
```

端口应可配置。

## 12.2 接口列表

### `POST /events`

用途：

接收 `cc-wrapper` 发来的标准事件。

请求体：

```json
{
  "event_id": "uuid",
  "event_type": "needs_input",
  "severity": "warning",
  "title": "Claude Code 等待你的输入",
  "message": "检测到当前任务暂停，等待你的继续操作。",
  "source": "cc-wrapper",
  "session_id": "20260410-090000-abc123",
  "process_id": 12345,
  "terminal_hint": {
    "window_title": "Claude Code",
    "cwd": "D:/project/demo"
  },
  "match": {
    "rule_id": "input.keyword.02",
    "pattern": "press enter",
    "sample_text": "..."
  },
  "timestamps": {
    "occurred_at": "2026-04-10T09:00:00+08:00",
    "sent_at": "2026-04-10T09:00:01+08:00"
  },
  "display": {
    "sticky": true,
    "play_sound": true,
    "timeout_ms": 0
  }
}
```

成功响应：

```json
{
  "ok": true,
  "accepted": true
}
```

### `GET /health`

用途：

1. 检查 `notifier-app` 是否在线
2. 供 `cc-wrapper` 启动前自检

成功响应：

```json
{
  "ok": true,
  "service": "notifier-app"
}
```

### `POST /focus`

用途：

1. 调试终端聚焦能力
2. 后续可给托盘菜单或按钮调用

请求体：

```json
{
  "window_title": "Claude Code",
  "cwd": "D:/project/demo"
}
```

## 12.3 接口约束

1. 只监听 `127.0.0.1`
2. 不做远程访问支持
3. 第一阶段不做鉴权
4. 但要校验请求体结构，避免错误数据导致 UI 崩溃

## 14. UI 设计要求

第一阶段 UI 不追求复杂，但必须保证清晰、稳定、可感知。

### 14.1 悬浮窗要求

1. 始终置顶
2. 尺寸固定或轻度自适应
3. 不抢占整个屏幕焦点
4. 明确显示标题、摘要、事件等级
5. 支持关闭
6. 支持点击“返回终端”

### 14.2 视觉层级建议

1. `info`：低刺激，冷静色
2. `warning`：明显但不过分刺眼
3. `critical`：高对比，立即引起注意

### 14.3 交互要求

1. 点击主体区域：尝试聚焦终端
2. 点击关闭按钮：关闭当前提醒
3. 若为 `sticky=false`，按超时自动消失
4. 若为 `sticky=true`，保持显示直到用户关闭或事件解除

## 15. 终端聚焦设计

这是第一阶段中技术上较不稳定的部分，需要接受降级。

### 15.1 目标

用户点击提醒窗后，系统尽量将焦点切回对应的 Windows Terminal 窗口。

### 15.2 候选实现

1. 通过窗口标题查找窗口
2. 通过进程信息查找窗口
3. 使用 Win32 API 将目标窗口置前

### 15.3 降级策略

如果无法准确聚焦：

1. 保留提醒窗
2. 给出提示文本，例如“未能自动切回终端，请手动切换”
3. 记录失败日志

## 16. 日志设计

第一阶段必须保留基础日志，方便规则调试。

### 16.1 wrapper 日志

记录：

1. Claude Code 启动命令
2. 进程 PID
3. 规则命中情况
4. 事件发送成功或失败
5. 进程退出码

### 16.2 notifier 日志

记录：

1. 服务启动状态
2. 收到的事件摘要
3. 弹窗动作
4. 声音播放动作
5. 终端聚焦成功或失败

### 16.3 日志原则

1. 不记录过长原始输出
2. 仅保留必要样本片段
3. 避免泄露敏感内容

## 17. 异常与降级处理

### 17.1 notifier 未启动

处理策略：

1. wrapper 启动时探测 `/health`
2. 若不可用，打印警告
3. 继续运行 Claude Code，不阻断主流程
4. 仅写本地日志

### 17.2 HTTP 发送失败

处理策略：

1. 进行有限次数重试
2. 若仍失败，则降级写日志

### 17.3 规则误报

处理策略：

1. 事件日志中记录 `rule_id`
2. 后续通过 `rules.yaml` 调整

### 17.4 UI 异常

处理策略：

1. 捕获异常并写日志
2. 尽量不让整个程序退出

## 18. 测试计划

第一阶段测试重点是“事件识别”和“提醒闭环”。

### 18.1 单元测试

至少覆盖：

1. 规则匹配
2. 去重逻辑
3. 空闲超时逻辑
4. 事件对象序列化
5. HTTP 接口基本校验

### 18.2 集成测试

模拟 Claude Code 输出流，验证以下场景：

1. 命中完成事件后是否弹提醒
2. 命中等待输入事件后是否弹提醒
3. 命中权限事件后是否弹提醒
4. 长时间无输出是否触发空闲提醒
5. 进程退出是否记录正确

### 18.3 手工验证

需要人工确认：

1. 悬浮窗位置是否合理
2. 置顶行为是否稳定
3. 点击后聚焦终端是否可用
4. 提醒是否会过度打扰

## 19. 里程碑拆分

建议按以下顺序实施：

### M1：基础骨架

目标：

## 20. 验收标准

第一阶段完成时，至少满足以下条件：

1. 用户可正常启动 `notifier-app`
2. 用户可通过 `cc-wrapper` 启动 Claude Code
3. 能识别并提醒以下事件：
   - 任务完成
   - 等待输入
   - 权限拦截
4. 提醒窗可在 Windows 上稳定置顶显示
5. 提醒不会因重复输出而疯狂弹出
6. 日志可用于排查误报和漏报
7. 在理想情况下，点击提醒可将用户带回终端

## 21. 开发注意事项

1. 第一阶段优先保证稳定，不追求功能过多
2. 规则设计要留可扩展空间，但不要提前做过度抽象
3. UI 和监听逻辑必须解耦
4. 所有事件都必须有唯一 ID 和时间戳
5. 优先把“能用”做出来，再做“更聪明”

## 22. 当前结论

第一阶段应聚焦于一个清晰的最小产品：

1. `cc-wrapper` 负责监听 Claude Code
2. `notifier-app` 负责桌面提醒
3. 通过本地 HTTP 传递标准事件
4. 用规则匹配实现事件识别
5. 用顶置悬浮窗完成提醒闭环

只要这个链路跑通，第二阶段再考虑历史导航与更深集成就会更稳。
