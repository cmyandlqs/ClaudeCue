# Claude Code 提醒助手方案设计

## 目标

基于当前需求，工具分为两个方向：

1. 高优先级：全局置顶桌面悬浮提示窗
2. 次优先级：终端侧边对话历史导航条

本阶段不直接确定唯一实现方案，而是先整理多个可验证路线，后续逐个做 PoC 验证。

## 需求理解

### 核心诉求

1. 当 Claude Code 在后台任务完成时，可以主动提醒用户回来查看
2. 当 Claude Code 遇到权限拦截、需要批准或需要继续交互时，可以及时唤醒用户
3. 最好能对历史对话进行节点化展示，并支持快速跳转定位

### 约束

1. 平台为 Windows
2. 使用环境为 Windows Terminal
3. 优先做高价值、低耦合、可快速验证的方案

## 方案拆分原则

建议将系统拆成两层：

1. 事件监听层：负责识别 Claude Code 是否需要提醒用户
2. 提示展示层：负责把提醒以悬浮窗、声音、托盘消息等形式展示出来

这样可以先把提醒体系做通，再逐步扩展到历史导航能力。

## 方案一：包装启动 Claude Code + 输出监听 + 悬浮窗

### 思路

不直接启动 Claude Code，而是通过一个包装启动器来启动。

包装器负责：

1. 启动 Claude Code 进程
2. 监听标准输出和标准错误
3. 识别关键事件
4. 通知一个常驻桌面程序弹出提醒

### 可识别事件

1. `task_completed`：任务完成
2. `needs_input`：需要用户输入
3. `permission_blocked`：权限拦截或等待批准
4. `error_detected`：异常中断
5. `idle_timeout`：长时间无输出但任务未结束

### 优点

1. 最容易落地
2. 不依赖 Windows Terminal 内部扩展能力
3. 适合先做 MVP
4. 事件识别逻辑后续可持续优化

### 风险

1. 依赖 Claude Code 的输出格式是否足够稳定
2. 部分事件可能需要通过规则推断，初期准确率有限
3. 需要用户通过包装器启动 Claude Code

### 验证重点

1. 是否能稳定捕获 Claude Code 输出
2. 是否能准确识别“完成 / 权限拦截 / 等待输入”
3. 提醒是否及时且不打扰

### 结论

这是最推荐的首个验证方案。

## 方案二：日志监听 / 输出落盘监听 + 悬浮窗

### 思路

如果 Claude Code 或其启动流程可以把输出写入日志文件，则可由后台程序直接监听日志变化，而不强依赖包装启动器。

### 实现方式

1. Claude Code 输出重定向到日志文件
2. 后台监听程序持续读取新增内容
3. 基于关键词或正则识别事件
4. 触发悬浮窗提醒

### 优点

1. 与 Claude Code 主启动流程耦合较低
2. 可以快速做 PoC
3. 调试方便，便于复盘识别规则

### 风险

1. 前提是日志内容完整、稳定
2. 某些交互态可能无法完整写入日志
3. 相比直接监听进程输出，实时性可能略差

### 结论

适合作为方案一的备选或过渡验证方案。

## 方案三：独立桌面常驻程序 + 托盘 + 顶置悬浮提醒

### 思路

无论事件来自方案一还是方案二，都统一交给一个独立常驻程序负责展示提醒。

### 建议功能

1. 系统托盘图标
2. 顶置悬浮窗
3. 提醒声音
4. 点击后聚焦回 Windows Terminal
5. 提醒级别区分

### 提醒级别建议

1. 普通完成：轻提示，自动消失
2. 需要输入：中强提醒，短暂停留
3. 权限拦截：高优先级提醒，置顶且更显著
4. 异常中断：高优先级提醒

### 优点

1. UI 与监听逻辑解耦
2. 以后可以接入更多事件源
3. 产品形态更完整

### 风险

1. 需要处理窗口置顶、焦点切换、任务栏行为等 Windows 细节
2. 提醒频率控制不好时会造成打扰

### 结论

这是推荐的展示层标准形态，应与方案一或方案二配合使用。

## 方案四：终端旁路历史导航窗

### 思路

不尝试直接嵌入 Windows Terminal，而是在终端旁边提供一个独立的伴随式导航窗。

该导航窗负责：

1. 将一轮会话拆成节点
2. 展示时间线或树状结构
3. 点击节点后进行快速定位

### 节点来源

1. 用户输入
2. Claude Code 回复
3. 工具调用
4. 错误事件
5. 权限请求
6. 阶段性完成节点

### 跳转方式候选

1. 在导航窗中打开节点详情
2. 通过关键词在终端中定位
3. 通过快捷键或自动输入辅助跳转

### 优点

1. 不依赖 Windows Terminal 官方扩展能力
2. 技术风险比“内嵌侧边栏”更低
3. 可以先验证“历史导航”是否真的有价值

### 风险

1. 终端内容与导航节点之间未必能精确双向映射
2. 真正的“点击跳到终端对应位置”实现难度可能偏高
3. 节点抽取需要较稳定的结构化依据

### 结论

如果要做历史导航，建议优先验证这一方案，而不是直接追求 Terminal 内嵌。

## 方案五：直接集成到 Windows Terminal 侧边栏

### 思路

尝试把历史导航能力直接做进 Windows Terminal 的窗口区域或作为深度集成插件。

### 优点

1. 用户体验理想
2. 视觉上更像原生能力

### 风险

1. Windows Terminal 是否提供足够合适的扩展机制需要先确认
2. 开发复杂度显著更高
3. 后续维护成本更高
4. 可能受 Terminal 更新影响

### 结论

这是长期探索项，不建议作为第一阶段目标。

## 推荐实施顺序

建议按以下顺序验证：

1. 先验证事件监听是否可行
2. 再验证桌面悬浮提醒体验是否足够好
3. 最后再评估历史导航是否值得继续投入

对应落地路径：

1. 方案一 + 方案三
2. 如果方案一受阻，则尝试方案二 + 方案三
3. 提醒能力稳定后，再验证方案四
4. 方案五仅作为长期备选

## MVP 建议

第一版只做最小闭环：

1. 启动 Claude Code 的包装器
2. 识别三类事件：
   - 任务完成
   - 等待输入
   - 权限拦截
3. 弹出顶置提醒窗
4. 支持点击后回到终端

先证明这个闭环有价值，再继续扩展。

## 技术选型建议

### 语言

优先建议 Python，原因：

1. 开发速度快
2. 适合快速做 PoC
3. 便于做进程监听、规则匹配和小型桌面程序

### UI

优先建议：

1. `PySide6`：更适合做正式桌面 UI
2. `tkinter`：更轻，但产品感较弱

如果目标是长期使用，建议优先 `PySide6`。

## 当前建议结论

当前最合理的主路线是：

1. 先做“Claude Code 监听器”
2. 再做“Windows 顶置悬浮提醒窗”
3. 历史导航先做独立伴随窗验证，不直接做 Terminal 内嵌

## 下一步建议

后续可以按以下节奏推进：

1. 明确 Claude Code 的可监听输入输出形式
2. 先写一个最小监听器原型
3. 写一个最小悬浮提醒窗原型
4. 将两者打通做第一版验证
