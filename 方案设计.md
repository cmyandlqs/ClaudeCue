# 方案设计

## 结论

项目正式放弃 wrapper 方案，后续仅保留 hook 方案。

## 为什么放弃 Wrapper

wrapper 方案依赖以下链路：

```text
wrapper -> 启动 claude -> 接管 stdout/stderr -> 规则匹配 -> 通知 UI
```

这条链路的问题不是实现细节，而是前提错误：

- Claude Code 需要真实 TTY
- 被 `PIPE` 接管后，交互行为失真或直接失败
- 即使局部功能能跑通，也不具备稳定性和可维护性

因此，继续优化 wrapper 没有投入价值。

## 新架构

```text
Claude Code 正常运行
  -> 官方 hooks 触发
  -> 本地 hook 脚本读取 stdin JSON
  -> 统一事件格式
  -> POST 到 notifier-app
  -> notifier-app 展示悬浮窗 / 托盘 / 声音 / 聚焦
```

## 组件拆分

### 1. Hook 层

职责：

- 接收 Claude Code hook payload
- 将不同 hook 事件映射成统一事件模型
- 调用本地 HTTP 接口

不负责：

- 长生命周期 UI
- 复杂状态持有
- 终端进程控制

### 2. Notifier 展示层

职责：

- 常驻后台
- 提供本地 HTTP 接口
- 展示悬浮提醒
- 托盘驻留
- 播放提示音
- 支持点击后聚焦回终端

### 3. 安装与配置层

职责：

- 写入或引导用户更新 `~/.claude/settings.json`
- 注册 `Notification`、`Stop` 等 hooks
- 提供 Windows 友好的一键配置脚本

## 推荐事件

第一阶段优先支持：

- `notification`
- `stop`
- `permission_required`
- `needs_input`

后续可扩展：

- `pre_tool_use`
- `post_tool_use`
- 更细粒度的任务阶段事件

## 统一事件模型建议

```json
{
  "event_id": "uuid",
  "event_type": "needs_input",
  "severity": "warning",
  "title": "Claude Code 等待输入",
  "message": "请回到终端继续交互",
  "source": "claude-hook",
  "session_id": "",
  "timestamps": {
    "occurred_at": ""
  },
  "display": {
    "sticky": true,
    "play_sound": true,
    "timeout_ms": 0
  }
}
```
